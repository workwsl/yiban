# 译伴（Yiban）PRD

## 1. 产品概述

本产品是一款浏览器插件，用于对网页内容进行双语对照翻译。用户在浏览外文网页时，可以通过插件将网页正文、标题、列表、表格、评论等内容翻译为目标语言，并保留原文显示。

产品参考“沉浸式翻译”的核心体验，但第一阶段聚焦更轻量、更可控的网页翻译能力，支持用户接入自己的大模型 API，例如 DeepSeek、通义千问 Qwen，以及兼容 OpenAI API 格式的模型服务。

## 2. 产品目标

### 2.1 核心目标

1. 帮助用户高效阅读外文网页。
2. 在不明显破坏原网页结构的前提下提供双语翻译。
3. 支持用户自定义模型服务，降低平台依赖。
4. 控制 API 成本，避免重复翻译。
5. 为后续 PDF、字幕、OCR、云同步等能力预留扩展空间。

### 2.2 非目标

1. 第一版不做账号系统。
2. 第一版不提供官方翻译额度。
3. 第一版不做云端同步。
4. 第一版不做移动端浏览器插件。
5. 第一版不做完整 PDF 翻译。

## 3. 目标用户

### 3.1 技术开发者

阅读英文技术文档、API 文档、GitHub、Stack Overflow、博客文章。重点需求是翻译准确、保留代码、保留变量名、保留命令行内容。

### 3.2 学生与研究人员

阅读论文页面、课程资料、英文百科、学术博客。重点需求是长文阅读、段落对照、术语稳定。

### 3.3 外贸与跨境用户

阅读海外商品页、供应商网站、行业报告。重点需求是网页整体理解、商品信息翻译、表格翻译。

### 3.4 普通外语阅读用户

阅读新闻、论坛、博客、社交媒体内容。重点需求是简单易用、自动翻译、低配置成本。

## 4. 使用场景

### 场景 1：手动翻译网页

用户打开英文网页，点击浏览器插件图标，点击“翻译当前页面”，网页中的可读文本被翻译并以双语方式显示。

### 场景 2：阅读技术文档

用户打开英文技术文档，插件翻译正文说明，但不翻译代码块、命令行、函数名、参数名和路径。

### 场景 3：选中文本翻译

用户只想翻译某一段内容，选中文本后出现浮动翻译按钮，点击后显示译文。

### 场景 4：设置网站规则

用户希望某个网站每次打开都自动翻译，例如英文文档站。用户可设置“总是翻译此网站”。

### 场景 5：切换模型

用户发现当前模型翻译效果不理想，可以在插件弹窗中切换 DeepSeek 或 Qwen 模型重新翻译。

### 场景 6：长页面滚动翻译

用户阅读很长的博客或文档页面，插件只翻译当前视口附近的内容，后续内容滚动到附近时再翻译。

## 6. 功能需求

### 6.1 网页翻译

用户点击“翻译当前页面”后，插件扫描当前网页中的文本节点，并将可翻译文本提交给模型 API。

翻译结果默认插入到原文下方：

```text
Original paragraph text...
译文段落内容...
```

译文样式要求：

1. 字号略小于或等于原文。
2. 颜色与原文区分，但不影响阅读。
3. 段落间距自然。
4. 不改变原网页主要布局。
5. 可通过设置调整译文字体大小和颜色。

网页翻译状态：

1. 未翻译
2. 翻译中
3. 已翻译
4. 已暂停
5. 翻译失败
6. 部分翻译失败

### 6.2 双语显示模式

MVP 默认只强制实现“双语对照”。后续支持三种模式：

1. 双语对照：原文和译文同时显示。
2. 仅译文：隐藏原文，只显示译文。
3. 悬浮译文：鼠标悬停或点击原文时显示译文。

MVP 可先在数据结构中预留 `displayMode` 字段。

### 6.3 选中文本翻译

用户选中网页文本后，页面附近出现一个小型浮动按钮。

点击后：

1. 获取选中文本。
2. 调用当前默认模型。
3. 在选区附近显示翻译浮层。
4. 用户点击其他位置后浮层消失。

浮层内容包括：

1. 原文，可折叠。
2. 译文。
3. 复制按钮。
4. 重新翻译按钮，可选。

### 6.4 模型配置

插件设置页需要支持添加模型服务。

模型配置字段：

```json
{
  "id": "deepseek-default",
  "provider": "deepseek",
  "name": "DeepSeek",
  "baseUrl": "https://api.deepseek.com",
  "apiKey": "",
  "model": "deepseek-chat",
  "enabled": true,
  "timeoutMs": 30000,
  "maxConcurrency": 3,
  "maxCharsPerRequest": 3000
}
```

首版支持：

1. DeepSeek
2. Qwen / 通义千问
3. OpenAI-compatible

用户能力：

1. 新增模型配置。
2. 编辑模型配置。
3. 删除模型配置。
4. 测试 API 是否可用。
5. 设置默认模型。

### 6.5 翻译提示词

系统需要内置默认提示词。

默认要求：

1. 只输出译文。
2. 不解释。
3. 不总结。
4. 不添加原文不存在的信息。
5. 保留 URL、邮箱、代码、变量名、命令行。
6. 保留 HTML 或 Markdown 中的结构含义。
7. 技术术语尽量准确。

可配置翻译风格：

1. 通用
2. 技术文档
3. 学术
4. 商务
5. 新闻
6. 自然口语

MVP 可以先实现“通用”和“技术文档”。

### 6.6 翻译策略

MVP 支持：

1. 手动翻译当前页面。
2. 网站黑名单。
3. 默认目标语言。

后续支持：

1. 自动检测网页语言。
2. 自动翻译非母语网页。
3. 指定网站总是翻译。
4. 指定网站从不翻译。
5. 指定网站使用特定模型。
6. 指定网站使用特定目标语言。

网站规则结构建议：

```json
{
  "domain": "example.com",
  "mode": "always_translate",
  "targetLanguage": "zh-CN",
  "modelId": "deepseek-default"
}
```

### 6.7 缓存机制

缓存目标：

1. 减少重复 API 调用。
2. 提升二次打开速度。
3. 降低用户模型成本。

缓存 key 建议由以下字段组成：

1. 原文文本 hash
2. 源语言
3. 目标语言
4. 模型 ID
5. 提示词版本
6. 翻译风格

缓存内容：

```json
{
  "key": "hash",
  "sourceText": "Original text",
  "translatedText": "译文",
  "sourceLanguage": "en",
  "targetLanguage": "zh-CN",
  "modelId": "deepseek-default",
  "createdAt": 1710000000000,
  "updatedAt": 1710000000000
}
```

缓存管理能力：

1. 启用 / 关闭缓存。
2. 清空全部缓存。
3. 按网站清理缓存，P1。
4. 设置缓存有效期，P1。

### 6.8 内容识别规则

需要翻译的内容：

1. `p`
2. `h1` - `h6`
3. `li`
4. `td`
5. `th`
6. `blockquote`
7. `article` 内文本
8. 评论区文本
9. 普通按钮文案，可选

不应翻译的内容：

1. `script`
2. `style`
3. `code`
4. `pre`
5. `kbd`
6. `samp`
7. `textarea`
8. `input`
9. `select`
10. `option`
11. URL
12. 邮箱
13. 纯数字
14. 过短文本
15. 不可见元素
16. 已翻译过的节点

技术文档页面需要重点保护：

1. 代码块
2. 行内代码
3. 文件路径
4. shell 命令
5. API 参数
6. JSON 字段名
7. 函数名
8. 类名

### 6.9 翻译队列

为了避免接口限流和页面卡顿，需要翻译队列。

队列要求：

1. 支持并发数控制。
2. 支持失败重试。
3. 支持暂停。
4. 支持停止。
5. 支持进度统计。
6. 支持去重。
7. 支持缓存命中跳过请求。

状态字段：

```json
{
  "total": 120,
  "translated": 80,
  "failed": 2,
  "pending": 38,
  "status": "translating"
}
```

### 6.10 错误处理

需要处理以下错误：

1. API Key 未配置。
2. API Key 无效。
3. 模型不存在。
4. 余额不足。
5. 请求超时。
6. 网络失败。
7. 接口限流。
8. 页面无可翻译内容。
9. 当前网站在黑名单中。
10. 用户主动停止翻译。

错误提示要求：

1. 简洁明确。
2. 告诉用户下一步怎么处理。
3. 不暴露完整 API Key。
4. 支持查看简短错误详情。

## 8. 隐私与安全

产品必须在设置页明确说明：

1. API Key 保存在用户本地浏览器。
2. 网页文本会发送给用户配置的模型服务商。
3. 插件自身不收集网页内容。
4. 插件不会翻译密码框和输入框内容。
5. 用户可以将敏感网站加入黑名单。
6. 导出配置时默认不导出 API Key，除非用户明确选择。

API Key 存储要求：

1. 使用浏览器扩展存储。
2. 不在页面 DOM 中暴露 API Key。
3. 不在 console 输出 API Key。
4. 请求错误中隐藏 API Key。

## 9. 非功能需求

### 9.1 性能

1. 普通文章页面开始翻译响应时间小于 1 秒。
2. 插件扫描页面不应造成明显卡顿。
3. 长页面使用分批处理。
4. 翻译请求并发默认不超过 3。
5. 单批文本默认不超过 3000 字符。

### 9.2 兼容性

MVP 支持：

1. Chrome 最新稳定版。
2. Edge 最新稳定版。

后续支持：

1. Firefox。
2. Safari。

### 9.3 稳定性

1. 插件异常不应导致页面崩溃。
2. 单个节点翻译失败不影响整体翻译。
3. 页面跳转后应停止旧页面翻译任务。
4. SPA 路由变化需要重新识别页面，P1。

## 10. 权限需求

浏览器插件可能需要以下权限：

1. `storage`：保存配置、缓存、网站规则。
2. `activeTab`：获取当前页面执行翻译。
3. `scripting`：注入内容脚本或执行页面翻译逻辑。
4. `contextMenus`：支持右键翻译选中文本。
5. `host_permissions`：允许访问用户配置的模型 API 域名。

权限原则：

1. 尽量少申请权限。
2. 不默认申请所有网站的高风险权限。
3. 对权限用途进行清晰说明。

## 12. 验收标准

MVP 验收条件：

1. 用户可以安装插件并打开设置页。
2. 用户可以配置 DeepSeek API Key 并测试成功。
3. 用户可以配置 Qwen API Key 并测试成功。
4. 用户打开英文网页后，可以点击按钮翻译当前页面。
5. 翻译后网页显示原文和中文译文。
6. 代码块不会被翻译。
7. 输入框内容不会被读取或翻译。
8. 同一段文本重复翻译时可以命中缓存。
9. 用户可以停止正在进行的翻译。
10. 用户可以选中文本并翻译。
11. API Key 错误时有明确提示。
12. 黑名单网站不会执行翻译。

## 14. 主要风险

1. 网页 DOM 结构复杂，可能导致翻译插入位置不稳定。
2. 大模型接口输出格式不稳定，可能影响批量文本拆分。
3. 长页面翻译成本较高。
4. API 限流会影响体验。
5. 部分网站 CSP 或动态渲染可能影响注入逻辑。
6. 技术文档中代码和正文边界需要谨慎处理。
7. 用户 API Key 的本地安全需要明确边界。

## 15. 后续需要细化的问题

进入技术方案前，还需要确认：

1. 第一版是否只做 Chrome，还是 Chrome + Edge 同步支持？
2. 是否需要完全兼容 OpenAI Chat Completions 格式？
3. Qwen 使用阿里云 DashScope 原生接口，还是 OpenAI-compatible 接口？
4. 缓存使用 `chrome.storage.local` 还是 IndexedDB？
5. 翻译结果是逐段请求，还是多段合并请求？
6. 是否需要内置默认网站规则，例如 GitHub、MDN、arXiv？
7. 插件 UI 使用原生 HTML/CSS，还是 React/Vue？
